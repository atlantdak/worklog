# Дизайн: приватный плагин `worklog` (конец дня → ClickUp)

> **Статус:** design approved 2026-06-05. Следующий шаг — план реализации (writing-plans).
> **Намеренно вне общих репозиториев** проектов: и плагин, и этот spec — приватные,
> чтобы механизм (ИИ + автоматизация воркло́га) не светился менеджменту и другим
> разработчикам. Наружу идёт только отполированный результат в ClickUp.

## 1. Назначение

Плагин конца рабочего дня. По словесному заданию пользователя собирает факты из
GitHub за окно, кладёт `.md`-черновики на сверку, после явного утверждения пишет
задачи в ClickUp через MCP и возвращает точные ссылки. **Агент ничего не пишет в
ClickUp до утверждения.** Постфактум-режим (а не планирование с утра) — выбран
пользователем.

«Фишка»: для коллег и менеджмента видны только аккуратные, дисциплинированно
ведённые задачи в ClickUp. *Как* они собраны (ИИ + этот плагин) — приватно.

## 2. Форма и приватность

- **Один переносимый плагин** с нейтральным именем `worklog` (без `alterpoka`/`ai`
  в названии). Структура как у `alterpoka-styleforge`: `command + skill + references
  + scripts`, плюс `.claude-plugin/plugin.json` и `marketplace.json`.
- **Хранение:** приватный GitHub-репозиторий; источник — `~/.claude/worklog/`.
  Глобальная установка `/plugin marketplace add <private-repo>` → доступен во всех
  проектах пользователя.
- **Секретно** = плагин + конфиг проекта + черновики. **Видно всем** = только задачи
  в ClickUp.
- В каждом рабочем (общем) репо: `.gitignore` на `worklog.config.json` и каталог
  черновиков (`_daily/` или заданный `drafts_dir`).
- **AlterPoka cleanup:** уже закоммиченный `ClickUp/tasks-ua/` удалить из общего репо
  (`git rm -r --cached`), оставить локально + добавить в `.gitignore`. Исходники
  черновиков не теряем, но из общей истории убираем.

## 3. Вход

Одна команда: `/log-day <scope>`. `<scope>` — обычные слова:

- дата: `вчера`, `2026-06-04`;
- PR: `#186`, `с #180`, `#180..#186`;
- пусто → агент спрашивает, какое окно брать.

Источник правды о том, *что брать* — слова пользователя. Никакого авто-«since last
run», никакого локального ledger.

## 4. Поток (4 стадии, 2 гейта)

- **S0 — Scope.** Разворачивает слова в конкретный набор коммитов/PR (через `gh`).
- **S1 — Сбор.** `gh` тянет PR/коммиты окна (номер, заголовок, даты, merge-статус,
  ссылка). Читает текущий лист ClickUp (ID задач, umbrella, уже упомянутые номера PR)
  — для дедупа и правильной привязки/ссылок. Координаты — из конфига проекта (§6).
- **S2 — Черновик → 🚦 ГЕЙТ 1.** Пишет `.md` в `drafts_dir/YYYY-MM-DD.md` в принятом
  формате (§7). Предлагает привязку (дописать в открытую задачу vs завести новую) —
  **только как предложение**. Прогоняет валидатор (§8). Показывает сводку + путь и
  **останавливается**. Сам в ClickUp не пишет.
- **S3 — Запись** (после явного «ок»). MCP создаёт/обновляет задачи: даты, статус,
  исполнитель, SP-заголовок, native-связи + ссылки в описании. `in progress → done`
  там, где PR смержен.
- **S4 — Ссылки.** Возвращает точные URL созданного/обновлённого.

## 5. Онбординг (первый запуск в проекте)

Если в проекте нет `worklog.config.json`: скилл проводит короткий init — берёт
`github_repo` из `git remote`, спрашивает ClickUp list / исполнителя / схему
нумерации / drafts_dir, пишет конфиг, добавляет его и drafts_dir в `.gitignore`.
Дальше ежедневные прогоны просто читают конфиг.

## 6. Конфиг на проект — `.claude/worklog.config.json`

```json
{
  "github_repo": "Growica/alterpoka",
  "clickup_list_id": "901217650623",
  "umbrella_task_id": "869djzk2m",
  "assignee_id": "93814769",
  "naming": { "scheme": "TASK-{n}", "sub": "TASK-{n}.{m}", "start_n": 21 },
  "sp_calibration": "~14-15 SP/активный день",
  "drafts_dir": "ClickUp/tasks-ua/_daily",
  "terminology": { "avoid": ["епік"], "use": ["задача", "під-задача"] },
  "language": "uk"
}
```

Вся специфика проекта — здесь, не в плагине. Соседний проект = свой конфиг, свой
лист, своя нумерация и язык.

## 7. references/ (generic, внутри плагина)

Только универсальное, без зашитых ID:

- `format.md` — шаблон описания задачи (`## 🎯 Story Points: N`, `↳ Огляд:`/
  `↳ Задача:` кликабельная ссылка, `📂 Під-задачі`, `🔗 Pull requests`), правило дат
  (`done → start+due`; `in progress → только start, без due`), дефолтная калибровка
  SP, запрет термина «епік» (или из `terminology` конфига).
- `worklog.config.example.json` — шаблон конфига.

## 8. scripts/ (машинные гейты)

- `collect-window.sh` — `gh` → JSON фактов окна (PR#, заголовок, даты, merge-статус,
  URL).
- dedup-helper — список уже залитых PR# из ClickUp для сверки на S1.
- `validate-draft` (`sh`/`php`) — суммы SP, правило дат, наличие обязательных
  заголовков/ссылок в черновике.

## 9. Намеренно НЕ делаем (YAGNI)

- Нет локального ledger / авто-«since last run» — окно задаёт пользователь словами.
- Нет автозаписи в ClickUp без гейта.
- Нет переименования/переписывания чужих (менеджерских) тикетов — максимум коммент.
- Нет планирования с утра — только постфактум.
- Нет привязки имени/идентичности к ИИ в публичных артефактах.

## 10. Открытые вопросы для плана реализации

- Язык скрипта-валидатора: `sh` (нет зависимостей) vs `php` (есть в стеке) — решить в
  плане.
- Формат хранения dedup-кэша на время одного прогона (только в памяти сессии vs tmp).
- Нужен ли `time tracking` ClickUp (start/stop) или достаточно start/due дат.
